Snapie Auth API

A purpose-built Hive account gateway and signing proxy. Authenticates users via Google or email, creates custodial Hive accounts, and signs blockchain operations on their behalf.

Base URLhttps://auth.snapie.io/api
ProtocolJSON over HTTPS
AuthSession cookie + CSRF token
Rate LimitsVaries per endpoint group

Overview

Snapie Auth bridges Web2 identity (Google, email) to Hive blockchain accounts. It handles the full lifecycle: account creation using pre-claimed tokens (ACTs), custodial key management, Hive operation signing, and voluntary key export (emancipation).

External services integrate via the Internal API using a shared Bearer token for account provisioning and sponsor token issuance.

Auth & Cookies

Sessions are stored in two cookies set by the server after any successful login:

CookiePurposeFlags
snapieauth_sessionJWT RS256 session tokenhttpOnly, Secure, SameSite=Lax
snapieauth_csrfCSRF double-submit token (JS-readable)Secure, SameSite=Lax

All state-mutating requests must include the CSRF token as the x-csrf-token header. Always send credentials: 'include' in fetch calls. There is no Bearer token for user-facing APIs.

Custody Modes

Custodial — Snapie generates all four Hive keypairs (owner, active, posting, memo) server-side, derived from a single master password via Hive's fromLogin. The master password and keys are AES-256-GCM encrypted using a server-derived key. The user never sees them until voluntary emancipation, when both the master password and all four keys are returned. All signing — posting and active — is handled transparently.

Emancipated — User holds their own keys. Snapie only has posting authority (granted by the user). Posting-level ops are still proxied. Active-level ops return an unsigned operation for the client to sign via Aioha or Keychain:

{ "needsClientSigning": true, "unsignedOp": ["op_name", {...}], "account": "alice", "keyType": "active" }

The account and keyType fields are always present so the consuming app can call Aioha or Keychain directly without additional context.

Error Codes

All errors return HTTP 4xx/5xx with body { "error": "error_code_string" }.

CodeStatusMeaning
unauthorized401No valid session cookie
admin_required403Endpoint requires isAdmin
csrf_invalid403Missing or wrong x-csrf-token header
already_linked409User already has a Hive account
previously_had_account403User already used their free account (sponsor token overrides)
global_daily_limit429Daily free account quota exhausted — resets midnight UTC
ip_daily_limit429Per-IP daily quota exhausted
no_acts503Snapie has no pre-claimed account tokens
invalid_username400Bad format or username already taken on-chain
insufficient_rc503Insufficient Resource Credits. For posting ops: the Snapie service account is out of RC (operator issue). For active ops: the user's own account is low.
emancipation_required403Account value exceeded threshold; user must export keys first
not_custodial400Action requires custodial mode
invalid_or_expired_challenge400Link challenge not found or expired (TTL 5 min)
invalid_signature401Hive posting key signature did not verify
posting_authority_required403@snapie does not have posting authority on the account
email_not_verified403Email address has not been verified yet
user_not_found404No account with that email (admin: grant admin)
cannot_demote_self400Admin tried to revoke their own access
op_not_allowed403Operation type is not permitted via the broadcast endpoint

POST /auth/apple

POST /auth/apple Public

Sign in or register with an Apple identity token. Creates a user record on first login. Sets session and CSRF cookies.

Apple quirk: iOS only provides the user's name and email on the very first sign-in. On all subsequent logins Apple returns neither — the server preserves whatever was saved. Your mobile app must pass name on the first login and omit it (or send null) on subsequent ones.

Request

FieldTypeDescription
identityTokenstringJWT from ASAuthorizationAppleIDCredential.identityToken
nameobject|nullOptional. { firstName, lastName } — only sent on first sign-in. Ignored if both fields are absent.

Response — { user: SnapieUser }

{
  "user": {
    "id": "6a272a4eac383450fbb0bc30",
    "name": "Alice Doe",
    "picture": null,
    "hiveUsername": null,
    "custodyMode": null,
    "isAdmin": false
  }
}

hiveUsername is null until the user creates or links a Hive account.

Environment variable

Set APPLE_CLIENT_ID to your app's Bundle ID (e.g. io.snapie.app). The server validates the aud claim in the identity token against this value.

React Native example

import { appleAuth } from '@invertase/react-native-apple-authentication'

const appleAuthRequestResponse = await appleAuth.performRequest({
  requestedOperation: appleAuth.Operation.LOGIN,
  requestedScopes: [appleAuth.Scope.EMAIL, appleAuth.Scope.FULL_NAME],
})

const { identityToken, fullName } = appleAuthRequestResponse
// fullName.givenName / familyName are only populated on first sign-in
const name = fullName?.givenName
  ? { firstName: fullName.givenName, lastName: fullName.familyName ?? undefined }
  : null

const res = await fetch('/api/auth/apple', {
  method: 'POST',
  credentials: 'include',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ identityToken, name }),
})
const { user } = await res.json()

POST /hive/sign-message

POST/hive/sign-messageSessionCSRF

Sign an arbitrary message with the user's Hive posting key. Custodial: server signs silently. Emancipated: returns the message for client signing. 30 req/min.

FieldTypeDescription
messagestringThe message to sign. Max 512 chars.
// Custodial
{ "signature": "1f3a...", "account": "alice" }

// Emancipated
{ "needsClientSigning": true, "message": "...", "account": "alice", "keyType": "posting" }

Signature is SHA-256 + secp256k1 (compatible with dhive/hive-js verifySignature).

Emancipated client signing:

// Keychain
window.hive_keychain.requestSignBuffer(account, message, 'Posting', cb)
// Aioha
aioha.signMessage(message, KeyTypes.Posting)

POST /hive/broadcast

POST/hive/broadcastSessionCSRF

Sign and broadcast a posting-level Hive operation using Snapie's posting authority. Works for all users (custodial and emancipated). 60 req/min.

Allowed op types: vote, comment, delete_comment, custom_json, claim_reward_balance.

FieldTypeDescription
oparrayA 2-element array: ["op_type", { ...fields }]
// Vote
{ "op": ["vote", { "voter": "alice", "author": "bob", "permlink": "my-post", "weight": 10000 }] }

// Root post
{ "op": ["comment", {
    "parent_author": "", "parent_permlink": "hive",
    "author": "alice", "permlink": "my-slug",
    "title": "My Post", "body": "Content here.",
    "json_metadata": "{\"tags\":[\"hive\"],\"app\":\"myapp/1.0\"}"
}] }

// Custom JSON
{ "op": ["custom_json", {
    "required_auths": [],
    "required_posting_auths": ["alice"],
    "id": "myapp",
    "json": "{\"action\":\"follow\",\"target\":\"bob\"}"
}] }

Response: { "txId": "abc123..." }

Note: json_metadata and json must be JSON-encoded strings, not objects.

POST /payment/hive-intent

POST/payment/hive-intentSession

Create a HIVE payment intent for account creation. The server generates a unique UUID memo. The user must send exactly the stated HIVE amount to the receiving account with that memo in the transfer. The server polls the Hive blockchain every 5 seconds and confirms the intent on detection. Rate limited to 10 req/hr.

No request body required.

{
  "memo": "a1b2c3d4-e5f6-...",
  "receivingAccount": "snapie",
  "amount": "3.000 HIVE",
  "amountUsd": 1.20,
  "expiresAt": "2026-06-12T16:30:00.000Z"
}

Send the transfer via Keychain or Aioha with memo as the transfer memo. Poll GET /payment/intent/:memo to track confirmation. Intent expires after 30 minutes.

POST/payment/lightning-intentSession

Create a Bitcoin Lightning payment intent via v4v.app. v4v converts sats to HIVE and forwards the payment to the operator's Hive account with the memo included. Rate limited to 10 req/hr.

No request body required.

{

  "invoice": "lnbc1234...",

Poll the status of a payment intent. Recommended interval: every 3–5 seconds.

POST /emancipate/start

POST/emancipate/startSessionCSRF

Decrypt and return all four private keys, plus the master password they derive from. Immediately wipes the encrypted copies from the server. Custodial accounts only. 5 req/hr. This action is irreversible.

Before returning keys, verifies them against on-chain public keys to guarantee correctness.

masterPassword is the single secret all four keys derive from (importable into any Hive wallet to regenerate them). It is null for accounts created before master-password support.

{
  "keys": {
    "owner":   "5J...",
    "active":  "5J...",
    "posting": "5J...",
    "memo":    "5J..."
  },
  "masterPassword": "P5J...",
  "message": "Save these keys now. They will not be shown again. This server has deleted its copy."
}
